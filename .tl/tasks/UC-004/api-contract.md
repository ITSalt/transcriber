# UC-004 — API Contract: Retry failed meeting processing

> Source: Neo4j SA layer (UC-004, RC-UC-004, RQ-034/035/036). Generated by /nacl-tl-plan --feature FR-001.
> Note: UC-004 has no `EXPOSES` APIEndpoint node in the graph; this contract is derived from
> the activity steps (UC-004-AS01..AS06) and RC-UC-004. Treat it as authoritative for dev.

## Endpoint

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| POST | `/api/meetings/:id/retry` | AUTHOR (own) | Re-enqueue the most recent FAILED job for the meeting and move the meeting back to its in-progress state. Idempotent on meetingId+stage. |

## Authentication / Authorization

- Role: **AUTHOR** (SR-01). Scope: `own` — caller must own the Meeting.
- SR-01 was granted `RU` (scope=own) on `TranscriptionJob` (ent-003) and `ProtocolGenerationJob` (ent-005) for this action.

## Request

No request body. The stage to retry is derived server-side from the most recent FAILED job (RQ-034).

```
POST /api/meetings/{id}/retry
```

## Response — 200 OK

```ts
// Returns the updated meeting (mirrors GET /api/meetings/:id shape)
interface RetryResponse {
  id: string;
  status: 'TRANSCRIBING' | 'PROTOCOL_GENERATING'; // chosen by failed stage (RQ-034)
  // ...rest of Meeting fields as per GET /api/meetings/:id
}
```

## Behaviour (from RC-UC-004 + activity steps)

1. Validate Meeting.status === `FAILED`. If not → 409 (no state change) — RQ-036.
2. Identify the most recent FAILED job (TranscriptionJob or ProtocolGenerationJob) — RQ-034.
3. In ONE Prisma transaction: reset that job (`status=QUEUED`, `attempt_count=0`, `error_msg=null`)
   AND transition `Meeting.status` FAILED → `TRANSCRIBING` (transcription failure) or
   `PROTOCOL_GENERATING` (protocol failure) — RQ-034, BRQ-008.
4. After commit: enqueue the job to BullMQ (idempotent on meetingId+stage; no duplicate if the
   stage job is already QUEUED/IN_PROGRESS — RQ-035) and publish SSE `meeting.status` on
   channel `meeting:<id>` — RQ-035.

## Errors

| Status | Code | When |
|--------|------|------|
| 409 | `MEETING_NOT_FAILED` | Meeting.status !== FAILED (RQ-036). No state change. |
| 409 | `RETRY_ALREADY_IN_FLIGHT` | A job for meeting+stage is already QUEUED/IN_PROGRESS — idempotent no-op (RQ-035). |
| 403 | `FORBIDDEN` | Caller is not the AUTHOR/owner of the meeting. |
| 404 | `MEETING_NOT_FOUND` | No such meeting for this owner. |

## Shared Types

`attempt_count: number` (Int) is added to `TranscriptionJob` and `ProtocolGenerationJob` DTOs by
TECH-026. The retry path resets it to 0. Zod schema additions land in `shared/`.

## Idempotency

Idempotency key = `meetingId + stage` (RC-UC-004). Double-click / concurrent retry is a safe no-op.
